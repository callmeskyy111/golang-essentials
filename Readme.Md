## Loops in GoLang ‚û∞
Let‚Äôs break down **loops in Go** in a clear and beginner-friendly way ‚Äî including **syntax**, **types**, **examples**, and **key differences** from other languages like JavaScript or C.

---

## üîÅ 1. What Are Loops?

In Go, **loops allow us to repeat a block of code** multiple times. Go has only **one loop construct**: `for`.

There is **no `while` or `do-while`** like in other languages ‚Äî everything is done using `for`.

---

## üü© 2. Basic `for` Loop

### Syntax:

```go
for initializer; condition; post {
    // loop body
}
```

### Example:

```go
for i := 0; i < 5; i++ {
    fmt.Println("i is", i)
}
```

* `i := 0` ‚Üí Initialization
* `i < 5` ‚Üí Condition
* `i++` ‚Üí Post-expression (increment)

This prints:

```
i is 0
i is 1
i is 2
i is 3
i is 4
```

---

## üîÑ 3. `for` as a While Loop

Go has **no `while` keyword**, but we can mimic it like this:

```go
i := 0
for i < 5 {
    fmt.Println(i)
    i++
}
```

This behaves exactly like a while loop in other languages.

---

## ‚ôæÔ∏è 4. Infinite Loop

We can create an infinite loop using just:

```go
for {
    fmt.Println("Looping forever...")
}
```

You‚Äôll usually combine this with `break` or `return` inside the loop to exit at some point.

---

## üõë 5. `break` and `continue`

* `break` ‚Üí Immediately exits the loop.
* `continue` ‚Üí Skips current iteration and moves to the next.

### Example:

```go
for i := 1; i <= 5; i++ {
    if i == 3 {
        continue // skip 3
    }
    if i == 5 {
        break // stop loop when i == 5
    }
    fmt.Println(i)
}
```

Output:

```
1
2
4
```

---

## üîÇ 6. Looping Over Collections (Array/Slice/Map)

### Over a slice:

```go
nums := []int{10, 20, 30}
for index, value := range nums {
    fmt.Println("Index:", index, "Value:", value)
}
```

### Over a map:

```go
m := map[string]int{"a": 1, "b": 2}
for key, value := range m {
    fmt.Println("Key:", key, "Value:", value)
}
```

You can also ignore unused variables with `_`:

```go
for _, v := range nums {
    fmt.Println(v)
}
```

## Error-Handling in GoLang ‚ö†Ô∏è
Let‚Äôs dive deep into **error-handling in GoLang** ‚Äî a very important and idiomatic part of Go programming.

---

## üß± **1. Go‚Äôs Philosophy on Errors**

Go does **not** use `try/catch/finally` like JavaScript, Python, or Java.

Instead, **errors are values**, returned as the **last return value** from functions, allowing **explicit and predictable error handling**.

> ‚úÖ Idiomatic Go: *"Don't panic. Handle errors explicitly."*

---

## üßæ **2. The `error` Type**

The `error` type is a built-in **interface** in Go:

```go
type error interface {
    Error() string
}
```

Any type that implements the `Error()` method (returning a string) satisfies the `error` interface.

---

## ‚öôÔ∏è **3. Returning and Handling Errors**

### A. Returning an error from a function:

```go
import "errors"

func divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, errors.New("cannot divide by zero")
    }
    return a / b, nil
}
```

### B. Handling the error:

```go
result, err := divide(10, 0)
if err != nil {
    fmt.Println("Error:", err)
    return
}
fmt.Println("Result:", result)
```

> ‚úÖ Go programmers check `if err != nil` **all the time**.

---

## üîß **4. Creating Custom Errors**

### A. With `errors.New`:

```go
import "errors"
err := errors.New("something went wrong")
```

### B. With `fmt.Errorf` (for dynamic messages):

```go
name := "Skyy"
err := fmt.Errorf("user %s not found", name)
```

---

## üß∞ **5. Custom Error Types (Advanced)**

You can define a struct that implements the `Error()` method.

```go
type DivideError struct {
    Numerator   float64
    Denominator float64
}

func (e *DivideError) Error() string {
    return fmt.Sprintf("cannot divide %.2f by %.2f", e.Numerator, e.Denominator)
}

func divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, &DivideError{a, b}
    }
    return a / b, nil
}
```

Now you can even **type-assert** or inspect error details!

---

## üß™ **6. Sentinel Errors (Comparing Errors)**

You might define reusable global error values:

```go
var ErrNotFound = errors.New("item not found")

func findItem(id int) error {
    return ErrNotFound
}

err := findItem(10)
if errors.Is(err, ErrNotFound) {
    fmt.Println("Handle item-not-found specifically")
}
```

Since Go 1.13+, use `errors.Is` and `errors.As`.

---

## ü™ú **7. `errors.Is` and `errors.As`**

### A. `errors.Is` ‚Üí Check if an error **matches** a known error

```go
if errors.Is(err, ErrNotFound) { ... }
```

### B. `errors.As` ‚Üí Check if error is of a specific **type**

```go
var myErr *DivideError
if errors.As(err, &myErr) {
    fmt.Println("DivideError with values:", myErr.Numerator, myErr.Denominator)
}
```

---

## üí• **8. `panic` and `recover` (Rare Use)**

Go also has `panic()` and `recover()` for unrecoverable issues ‚Äî like nil dereference or failed assertions.

### A. `panic()` ‚Üí Immediately stops execution and unwinds the stack

```go
panic("something went terribly wrong")
```

### B. `recover()` ‚Üí Used to catch a panic **only inside a `defer`**

```go
func mayPanic() {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("Recovered from panic:", r)
        }
    }()
    panic("Boom üí•")
}
```

‚úÖ Only use `panic` for unrecoverable errors (e.g., corrupted memory or developer mistakes).

---

## üß† Best Practices in Go Error Handling

| Practice            | Description                           |
| ------------------- | ------------------------------------- |
| `if err != nil`     | Always check errors right away        |
| `return nil`        | Return nil if no error occurred       |
| `fmt.Errorf`        | For contextual messages               |
| `errors.Is/As`      | For error comparison and inspection   |
| Don‚Äôt use panic     | Except for truly unexpected errors    |
| Don‚Äôt ignore errors | Compiler may allow it, but it‚Äôs risky |

## üîç **Difference Between `panic` and `error` in Go**

| Feature                  | `error`                                                   | `panic`                                                            |
| ------------------------ | --------------------------------------------------------- | ------------------------------------------------------------------ |
| üß† Concept               | Expected issues                                           | Unexpected, critical failures                                      |
| üõ†Ô∏è Used for             | Recoverable problems (e.g. file not found, invalid input) | Unrecoverable problems (e.g. array out-of-bounds, nil dereference) |
| üîÅ How it works          | Returned as a value                                       | Stops normal flow, starts unwinding                                |
| üéØ Goal                  | Let the caller decide what to do                          | Exit the program or recover via `recover()`                        |
| üß™ Should be tested      | Yes ‚Äî check `err != nil`                                  | Rarely ‚Äî usually you avoid triggering `panic` in tests             |
| üßò Recoverable?          | Yes                                                       | Only inside a `defer` using `recover()`                            |
| üî• Used like exceptions? | No                                                        | Somewhat, but only for **severe** issues                           |

## üßµ `error`: Normal Flow of Handling Issues

* **Idiomatic Go** always prefers **returning errors** instead of throwing them.
* You get the error as a value and decide what to do with it.

### ‚úÖ Example: Using `error`

```go
func divide(a, b float64) (float64, error) {
	if b == 0 {
		return 0, fmt.Errorf("cannot divide by zero")
	}
	return a / b, nil
}

func main() {
	result, err := divide(10, 0)
	if err != nil {
		fmt.Println("Handled error:", err)
		return
	}
	fmt.Println("Result:", result)
}
```

---

## üí• `panic`: For Critical Failures

* A `panic` **immediately stops execution** and **starts unwinding the call stack**.
* You can **recover** using `recover()` inside a `defer` block, but this is rare and mainly used in **library code**, **servers**, or **frameworks**.

### ‚ùå Example: Using `panic` (Not idiomatic in most cases)

```go
func divide(a, b float64) float64 {
	if b == 0 {
		panic("division by zero")
	}
	return a / b
}

func main() {
	fmt.Println("Result:", divide(10, 0)) // üî¥ program will crash here
}
```

Output:

```
panic: division by zero
```

---

## üßò‚Äç‚ôÇÔ∏è `recover`: Catching Panics

You can **recover** from a panic to keep your app alive:

```go
func safeDivide(a, b float64) {
	defer func() {
		if r := recover(); r != nil {
			fmt.Println("Recovered from panic:", r)
		}
	}()
	fmt.Println("Result:", a/b)
}

func main() {
	safeDivide(10, 0)
	fmt.Println("App is still running...")
}
```

## üß† When to Use What?

| Use Case                                      | Use `error` ‚úÖ | Use `panic` ‚ö†Ô∏è |
| --------------------------------------------- | ------------- | -------------- |
| File not found                                | ‚úÖ             | ‚ùå              |
| User input invalid                            | ‚úÖ             | ‚ùå              |
| Failed to connect to DB (known issue)         | ‚úÖ             | ‚ùå              |
| Nil pointer dereference (bug in code)         | ‚ùå             | ‚úÖ              |
| Array out-of-bound (programmer mistake)       | ‚ùå             | ‚úÖ              |
| Unexpected internal corruption or assert fail | ‚ùå             | ‚úÖ              |
| Library should force crash on misuse          | ‚ùå             | ‚úÖ              |
| Critical init failure (e.g. missing ENV)      | ‚ùå             | ‚úÖ (sometimes)  |

---

## üîê TL;DR

* ‚úÖ Use `error` for **normal, expected** failures.
* ‚ö†Ô∏è Use `panic` for **unexpected, unrecoverable** issues (like bugs).
* üõë Never use `panic` to control logic or skip normal error handling.
* üßò Only use `recover()` in very controlled situations ‚Äî e.g., web servers to prevent crashing the entire app.

## üîê Linux File Permissions: Overview

In Linux, every file and directory has **permissions** associated with it that define **who can read, write, or execute** them. These permissions are set for three kinds of users:

### üë• User Types

| Symbol | Description                                 |
| ------ | ------------------------------------------- |
| `u`    | User (Owner) ‚Äî the person who owns the file |
| `g`    | Group ‚Äî a set of users in the same group    |
| `o`    | Others ‚Äî everyone else on the system        |
| `a`    | All (user + group + others)                 |

---

## üî§ Permission Types

Each user type can have three kinds of permissions:

| Symbol | Name    | Meaning on File            | Meaning on Directory        |
| ------ | ------- | -------------------------- | --------------------------- |
| `r`    | Read    | Can view file contents     | Can list directory contents |
| `w`    | Write   | Can modify file contents   | Can add/remove files in dir |
| `x`    | Execute | Can run the file as script | Can `cd` into the directory |

---

## üßæ Viewing File Permissions (`ls -l`)

```bash
ls -l myfile.txt
```

Output example:

```
-rwxr-xr--
```

Breakdown:

* `-` : Type (dash = regular file, `d` = directory)
* `rwx` : Owner has **read, write, execute**
* `r-x` : Group has **read, execute**
* `r--` : Others have **read only**

---

## üî¢ Numeric (Octal) Representation

Each permission is represented by a number:

* `r = 4`
* `w = 2`
* `x = 1`

Sum them up for each user group:

| Permissions | Value |
| ----------- | ----- |
| `rwx`       | 7     |
| `rw-`       | 6     |
| `r--`       | 4     |
| `---`       | 0     |

So:

```bash
chmod 755 myfile.sh
```

Means:

* Owner: `7` ‚Üí `rwx`
* Group: `5` ‚Üí `r-x`
* Others: `5` ‚Üí `r-x`

---

## üõ†Ô∏è Changing Permissions

### `chmod` ‚Äì Change mode

```bash
chmod 644 file.txt   # rw-r--r--
chmod +x script.sh   # adds execute permission
chmod u+x script.sh  # adds execute only for owner
```

### `chown` ‚Äì Change ownership

```bash
chown skyy:developers file.txt
# User = skyy, Group = developers
```

---

## üè∑ Special Permissions (Advanced)

| Symbol | Name          | Description                                      |
| ------ | ------------- | ------------------------------------------------ |
| `s`    | setuid/setgid | Executes with owner/group privileges             |
| `t`    | sticky bit    | Only owner can delete files (commonly on `/tmp`) |

Example: `chmod 4755 file` sets **setuid** (4 before permissions).

---

## üîê Typical Use-Cases

| Use Case                    | Command                 |
| --------------------------- | ----------------------- |
| Make a script executable    | `chmod +x script.sh`    |
| Give read/write to everyone | `chmod 666 file.txt`    |
| Secure a private file       | `chmod 600 secrets.txt` |
| Make directory accessible   | `chmod 755 /myfolder`   |
| Make file owner-only        | `chmod 700 private.sh`  |

## üßë‚Äçüíª Scenario: Deploying a Go Web App on a Linux Server

Let's say we have a Go app with the following structure:

```
/home/skyy/my-go-app/
‚îú‚îÄ‚îÄ main.go
‚îú‚îÄ‚îÄ app        # compiled binary
‚îú‚îÄ‚îÄ config.env
‚îî‚îÄ‚îÄ static/    # static files (HTML, CSS, JS)
```

We want:

* Only **Skyy** (the owner) to run and modify the app.
* The server (e.g., Nginx or systemd) to **execute** the Go binary.
* The public to access only what's needed (like `static/` files if served directly).
* **No one else** (other users) to view configs or tamper with code.

---

## ‚úÖ Step-by-Step File Permission Setup

### 1. **Binary (`app`)**

```bash
chmod 755 app
```

Means:

* `7` for owner (read/write/execute)
* `5` for group (read/execute)
* `5` for others (read/execute)

> ‚úÖ Allows executing the app, but **only the owner can modify** it.

### 2. **Main Go Source File (`main.go`)**

```bash
chmod 644 main.go
```

* Owner: read/write
* Group/Others: read

> ‚úÖ Useful during dev; in production you might remove it or set it to `600`.

### 3. **Environment Config (`config.env`)**

```bash
chmod 600 config.env
```

> üîí **Very important**: Only the owner can read/write it. Prevents exposing secrets.

### 4. **Static Assets (`static/`)**

```bash
chmod -R 755 static/
```

* `-R`: recursively sets permissions

> ‚úÖ Allows public read access for web server to serve files, but no write access.

---

## üîê Ownership Control (Optional)

If your Go app runs as a different Linux user (e.g., via a systemd service), you might:

```bash
chown root:www-data app
chmod 750 app
```

> This allows the app to be run by `www-data` (like Nginx), but **not by everyone**.
